## 1. 문서 목적

Feedback Analyzer는 **자연어 고객 피드백을 수집·키워드 분류·감정 분석·시각화**하는 Spring Boot + Thymeleaf 웹 앱이다.  
초기 버전은 의도적으로 레거시·코드 스멜을 포함한다.

본 문서는 **기능 명세를 되묻는 것이 아니라**, 실제 업무·과거 행동·비용·실패 사례를 끌어내어 **진짜 문제와 가정**을 고정하기 위한 Mom Test 질문 10개이다.

### 1.1. Mom Test 원칙 (본 프로젝트 적용)

| 하지 말 것 | 할 것 |
|------------|--------|
| "이 기능 필요하세요?" | "지난주에 비슷한 일을 어떻게 끝냈어요?" |
| "CSV 업로드 좋을 것 같은데요" | "마지막으로 대량 피드백을 넘길 때 파일 형식이 뭐였어요?" |
| 가상의 미래 시나리오만 | 구체적 일시·건수·문장·도구 이름 |
| 칭찬·의견 수집 | 의사결정·다음 행동·포기한 이유 |

### 1.2. 네 가지 검증 축 (질문 전체에서 반복 등장)

| 축 | 코드/PRD 연결 | 검증하고 싶은 가정 |
|----|---------------|-------------------|
| **(1) 피드백 수집** | FR-02, FR-03, `Session`, CSV `text` | "웹에 직접 입력·CSV면 충분하다" |
| **(2) 키워드 분류** | FR-05~06, `Constants`, `Filters` | "5개 카테고리·contains 매칭이면 운영에 쓸 만하다" |
| **(3) 감정 분석** | FR-04, FR-09, `TextAnalyzer` vs `Filters` | "긍정/부정/중립 3분류면 리포트에 쓸 수 있다" |
| **(4) 결과 시각화** | FR-07~08, `index.html`, `fil_data` | "건수 통계·CSV 다운로드면 분석 업무가 끝난다" |

## 2. Mom Test 질문 10개

> 각 질문 끝의 **[축]** 은 주로 깊게 파고드는 영역이다. 한 질문이 여러 축을 자연스럽게 건드릴 수 있다.

---

### Q1. 수집 경로의 실체

**질문**  
"가장 최근에 고객 불만·칭찬 문장을 **한곳에 모은** 경험을 떠올려 주세요. 그때 데이터는 처음에 **어디에** 쌓였고, 분석 담당자에게 **어떤 형태**로 넘겼나요? (티켓 번호, 날짜, 채널이 같이 있었는지도요.)"

**[축] (1) 수집**  
- 검증 가정: "텍스트 한 컬럼 CSV + 웹 입력이 실제 수집 파이프라인과 맞다."  
- 좋은 답의 신호: 티켓 시스템·메신저·설문 등 **원천이 여러 곳**이고, CSV는 **2차 가공물**이라는 설명.  
- 나쁜 답의 신호: "그냥 CSV 쓸 것 같아요" (경험 없음).

---

### Q2. 수집 직후의 첫 번째 행동

**질문**  
"피드백 파일을 받은 **당일**, 분석 전에 **가장 먼저** 하신 작업 한 가지는 무엇이었나요? (예: 중복 제거, 욕설 마스킹, 날짜 붙이기, 담당자 배분 등)"

**[축] (1) 수집 → (4) 시각화 전처리**  
- 검증 가정: "업로드 즉시 감정·키워드 집계가 1순위 니즈다."  
- `01_analysis.md`: 업로드 후 통계 미표시·`C:\tmp` 등 **수집 직후 UX** 갭이 실제 니즈인지 판별.

---

### Q3. 키워드 분류의 실패 사례

**질문**  
"**'배송'**처럼 뻔한 단어로 묶었을 때 **놓쳤던** 민원이 있다면, 기억나는 **문장 하나**만 말해 주실 수 있을까요? 그때 왜 기존 분류로는 부족했나요?"

**[축] (2) 키워드 분류**  
- 검증 가정: "main 키워드 5카테고리 + contains면 충분하다."  
- Shotgun Surgery·`Constants`/`Filters` 이중 정의가 **운영 비용**인지, **기능 부족**인지 구분.

---

### Q4. 카테고리 변경 빈도와 주체

**질문**  
"지난 1년 안에 '이건 배송이 아니라 품질이다'처럼 **분류 기준을 바꾼** 적이 있나요? 있다면 **누가**, **얼마나 자주**, **무엇을** 기준으로 바꿨나요?"

**[축] (2) 키워드 분류 → (7단계) File DB**  
- 검증 가정: "하드코딩 `Constants`로도 운영 가능하다" vs "File DB·설정 UI가 필수다."  
- PRD FR-18 우선순위 근거.

---

### Q5. 감정 라벨로 한 일

**질문**  
"**긍정/부정 비율** 숫자를 보고 **실제로 조치**한 사례가 있나요? 있다면 그 숫자를 **누구에게**, **어떤 결정**에 썼고, **없었다면** 왜 그 숫자만으로는 부족했나요?"

**[축] (3) 감정 분석**  
- 검증 가정: "3분류 건수 막대만으로 PM·CS가 의사결정한다."  
- `TextAnalyzer` vs `Filters` **중립 불일치**가 실무에서 치명적인지(리포트 신뢰 붕괴).

---

### Q6. ‘중립’의 업무적 의미

**질문**  
"운영하실 때 **'중립'** 피드백은 보통 **무시**하시나요, **별도 큐**로 보시나요? 최근 한 달에 중립으로 분류됐는데 **나중에 부정으로 바뀐** 문장이 있었나요?"

**[축] (3) 감정 분석**  
- FR-09(중립 필터 버그)의 **비즈니스 심각도** 판단.  
- `괜찮` 등 키워드 중복·우선순위 이슈(`01_analysis.md` P0-1)가 **실제 분류 논쟁**으로 이어지는지.

---

### Q7. 감정·키워드 교차 조회

**질문**  
"**'배송 + 부정'**처럼 감정과 주제를 **동시에** 거른 적이 있나요? 그때 **몇 건**이 나와야 '이번 주 이슈'로 올렸고, 결과는 **어디에** 적어 두셨나요?"

**[축] (2)(3) → (4) 시각화·다운로드**  
- `POST /filter`, `fil_data`, download 연계.  
- 필터 없이 다운로드·빈 CSV(`01_analysis.md` P1-3)가 **업무 중단**으로 이어지는지.

---

### Q8. 화면 vs 엑셀 재작업

**질문**  
"웹이나 대시보드에서 통계를 본 뒤에도 **결국 엑셀(또는 슬라이드)로 다시 옮긴** 적이 있나요? 있다면 **어떤 숫자·표**가 빠져서였나요?"

**[축] (4) 시각화**  
- 검증 가정: "건수 stat-box 시각화 = 분석 완료."  
- Trend(FR-17)·피드백 목록 미표시(`index.html`) 필요성.

---

### Q9. 공유·보고 단위

**질문**  
"분석 결과를 **팀 밖**(경영진·개발·물류 등)에 공유할 때 **마지막으로 보낸 산출물**은 무엇이었나요? 파일 형식·주기·**원문 문장 포함 여부**를 말해 주세요."

**[축] (4) 시각화 + (1) 수집**  
- CSV 다운로드·UTF-8 BOM·헤더 오류(`downloadFile`)의 실무 영향.  
- "통계만" vs "원문 + 라벨" 니즈 분리.

---

### Q10. 포기·우회한 순간

**질문**  
"자동 분류·감정 태깅을 **시도했다가 포기**하거나, **사람이 직접** 읽기로 돌아간 적이 있나요? 그때 **결정적 이유** 한 가지와, 그때 쓰던 **도구**를 알려 주세요."

**[축] (1)(2)(3)(4) 통합**  
- 제품 전체 가치 vs **리팩토링 실습** 목적 분리.  
- 테스트 90%·로그 UI(FR-11)가 **운영 신뢰**에 기여하는지.

## 4. 코드 스멜 연계 사례 3건

Mom Test에서 위 질문으로 드러날 수 있는 **초기 버전 문제**를, `project_purpose.md` §4·`01_analysis.md`의 스멜과 연결한다.  
(구현·수정 **금지** — 인식·리팩토링 백로그용.)

---

### 사례 1 — 감정 규칙 이중화 · 중립 필터 신뢰 붕괴

- **Mom Test 연결**: Q5, Q6, Q7 — "부정 비율로 보고했는데 현장에서는 중립/부정이 섞여 논쟁" 또는 "중립만 필터하면 절반 사라짐."
- **드러나는 스멜·안티패턴**
  - **중복 코드**: `TextAnalyzer.sent()` vs `Filters.S_KEYWORDS` vs `Constants.SENTIMENT_KEYWORDS`에 감정 판별 분산
  - **Shotgun Surgery**: 중립 키워드 수정 시 여러 파일 동시 변경
- **초기 버전 증상** (`01_analysis.md` P0-1): 동일 문장이 분석 집계와 필터 결과에서 **다른 감정**; `괜찮`이 긍정·중립에 중복
- **리팩토링 방향 (문서만)**: 단일 `SentimentClassifier` + FR-09 AC — **외부 계약(URL) 불변**

---

### 사례 2 — 수집·다운로드 단절 · God Controller

- **Mom Test 연결**: Q2, Q7, Q9 — "필터한 목록만 경영진에 보냄" / "다운로드 눌렀는데 빈 파일" / "업로드 후 바로 통계가 안 보임."
- **드러나는 스멜·안티패턴**
  - **God Function / God Object**: `FeedbackController`가 CSV 파싱·`C:\tmp`·`fil_data`·다운로드 헤더까지 담당
  - **전역 변수**: `fil_data`는 `/filter` 후에만 채움 — 분석만 한 사용자와 **데이터 소스 불일치**
  - **Lava Flow**: `FileHandler` 주입만 되고 수집·저장 경로와 **미연결**
- **초기 버전 증상**: `Content-Disposition` 오류, 업로드 후 `sentimentResults` 미전달, Windows 전용 임시 경로
- **리팩토링 방향 (문서만)**: `FeedbackService` + 명시적 "보내기 대상" 모델 — FR-03, FR-08

---

### 사례 3 — 키워드·카테고리 운영 비용 · 가짜 도메인 모델

- **Mom Test 연결**: Q3, Q4, Q10 — "배송 아닌데 배송으로 묶임" / "분류 기준을 분기마다 바꿈" / "결국 사람이 다시 읽음."
- **드러나는 스멜·안티패턴**
  - **매직 스트링·하드코딩**: `Constants.CATEGORY_KEYWORDS` + `UIComponents.CATS` 이중 유지
  - **부적절한 네이밍**: `fil`, `kw` — 운영 규칙을 코드에서 찾기 어려움
  - **데이터 클래스 미활용**: `Feedback.sentiment`/`category` 필드 미설정 → 분류 결과가 **UI·다운로드에 재사용 불가**
  - **Feature Envy**: `Filters`가 `CATEGORY_KEYWORDS` 내부 `sub` Map 구조에 강결합
- **초기 버전 증상**: 카테고리 1개 추가 시 4~5파일 수정; 키워드 DB화(FR-18) 없이는 Q4 답과 **불일치**
- **리팩토링 방향 (문서만)**: enum/설정·File DB·`Feedback`에 분석 결과 저장 — FR-12~13, FR-18
