
## docs 폴더 산출물 명명 규칙
 : 작업 산출물을 해당 폴더에 저장함.
- 형식: `{순번2자리}_{제목}.md` (예: `00_analysis.md`)
- 순번: `docs/` 기존 파일 최대 번호 + 1

## report 폴더 산출물 명명 규칙
: 작업에 대한 결과 보고서를 작성해서 해당 폴더에 저장함.
- 형식: `{순번2자리}_{제목}.md`
- 순번: `report/` 기존 파일 최대 번호 + 1

## prompting 폴더 산출물 명명 규칙
: 작업 단계별로 agent 와의 대화 전체를 파일로 생성.
    - 형식: `{순번2자리}_{제목}.md`
    - 순번: `prompting/` 기존 파일 최대 번호 + 1
: 추가적으로
    - User_prompt.md 는 user 가 입력한 prompt 만 표 형식으로 저장. 요약본도 자동 생성
    - GIT_prompt.md 는 anent 와의 대화중 git 명령어만 추출하여 저장. 요약본에 명령어가 중복되지 않고 사용된 명령어만 정리

##  Feedback Analyzer 프로젝트에 대한 개발 전략

1. 문제 코드 및 기본 요구사항 분석 
   - 기본 코드구조, 로직 이해
2. 기본 요구사항 및 품질 요구사항 구현 
   - OCP를 만족하는 인터페이스 구현 
   - SRP를 만족하도록 클래스 구현 
   - 입력값 검증을 위한 구현
3. TC 구현
   - 단위변환 기능 검증 및 입력 값 검증 TC 작성 
4. 추가 요구사항 구현 
   - 3개 요구사항 구현 및 TC 작성 
5. 회고 및 발표 
   - 실습 목표와 달성도
   - AI를 어떻게 활용했나? 도움이 된 순간과 한계는?
   - TC를 추가해보면서 개선에 미친 영향, TC 작성 팁
   - 클린코드와 리팩토링에서 느낀 장점과 어려운점

## GIT 브랜치 구조
main
 └── A_01                     ← 통합 브랜치 (main에서 생성)
      ├── spec                 → PR → A_01 머지 
      ├── red                  → PR → A_01 머지 
      ├── green                → PR → A_01 머지 
      ├── refactoring          → PR → A_01 머지 
      └── new_feature  → PR → A_01 머지 

최종: A_01 → main  (모든 단계 완료 후 릴리스 PR, 단 각 브랜치들은 삭제하지 않음)
