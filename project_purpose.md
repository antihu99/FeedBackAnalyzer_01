# 📄 프로젝트 목적

## 1. 프로젝트 개요

### 1.1. 프로젝트명
**리팩토링 챌린지: 고객 피드백 분석 시스템 (Feedback Analyzer)**

### 1.2. 프로젝트 목적
이 프로젝트는 자연어 기반 고객 피드백 데이터를 수집, 분류, 시각화하는 기능을 제공하는 Springboot + thymeleaf 기반 애플리케이션입니다. 초기 버전은 구조화되지 않은 코드, 중복 로직, 잘못된 책임 분배 등 다양한 코드 스멜을 포함하며, 학습자는 이를 직접 식별하고 리팩토링을 통해 클린 아키텍처로 개선해보는 경험을 할 수 있습니다.

### 1.3. 주요 학습 목표
- 코드 스멜 및 안티패턴 식별 능력
- MVC 또는 레이어드 아키텍처로 구조 개선
- 비즈니스 로직과 UI 분리
- 모듈화, 클래스 재구성, 추상화 등 다양한 리팩토링 기법 실습
- Springboot 기반 프론트엔드 유지보수성 향상
- 테스트 가능한 아키텍처 구성
- 상태 관리 및 의존성 주입 설계 경험

### 1.4. 대상 학습자
- 중급 이상 Java 개발자
- 복잡한 구조를 리팩토링하고 싶어하는 학습자
- TDD, Clean Code, Refactoring 관련 실습이 필요한 개발자

## 2. 초기 기능 명세 (리팩토링 전 상태)

### 2.1. 핵심 기능 목록
- 웹 어플리케이션 대시보드 제공
- 텍스트 피드백 입력 (수동/CSV 업로드)
- 간단한 키워드 기반 피드백 분류
- 기본 감정 분석 (긍정/부정/중립)
- 피드백 필터링 및 검색 기능
- 분류 결과 시각화

### 2.2. 사용자 시나리오
- 사용자가 대시보드에 접속(http://localhost:8080)
- 피드백을 입력하거나 파일을 업로드
- 시스템이 키워드 및 감정 기반으로 피드백 분류
- 사용자가 특정 키워드나 감정 기준으로 결과 필터링
- 통계 결과를 확인하고, 결과를 CSV로 다운로드


### 2.3. 입력/출력 명세

| 기능               | 입력 예시                           | 출력 예시                                   |
|--------------------|--------------------------------------|----------------------------------------------|
| 텍스트 입력        | "배송이 너무 늦어요. 화가 납니다."    | 감정: 부정, 카테고리: 배송                   |
| CSV 업로드         | `feedbacks.csv` (텍스트 컬럼 포함)   | 전체 피드백 분석 결과 테이블 및 그래프      |
| 감정 분석 결과     | -                                    | 긍정: 40%, 중립: 20%, 부정: 40%              |
| 키워드 필터링      | "배송"                                | 배송 관련 피드백 15건 출력                   |

## 3. 초기 코드 구조 설계 (리팩토링 전 상태)

### 3.1. 파일 구성 (총 12개 파일)

| 파일명               | 역할 및 특징                                                             |
|----------------------|---------------------------------------------------------------------------|
| `DemoApplication`    | Springboot 앱 시작점 (`mvn springboot:run`)                                |
| `FeedbackController` | Springboot 앱 Controller 모음 ( 데이터 로직과 혼합 )                       |
| `TextAnalyzer`       | 감정 분석, 키워드 추출 기능 (중복 많음)                                     |
| `FileHandler`        | CSV 파일 읽기, 저장 로직 (데이터 로직과 UI 혼합)                            |
| `Constants`          | 감정 키워드, 분류 키워드 하드코딩                                           |
| `Filters`            | 필터링 로직 구현 (데이터 흐름 불분명)                                       |
| `UIComponents`       | UI Category만 존재 (로직과 혼재)                                            |
| `Logger`             | 로그 출력(print 남용), 사용처 불분명                                        |
| `Session`            | 상태 관리용 전역 변수들 정의                                                |
| `DemoApplicationTests`| 테스트 흉내만 낸 파일, 실제 테스트는 없음                                  |
| `index.html`         | 키워드 필터등 불분명한 UI 연결                                              |


## 4. 의도적으로 포함할 코드 스멜 및 안티패턴

### 4.1. 코드 스멜 목록 및 예시

| 스멜 유형              | 포함 예시                                                                 |
|------------------------|--------------------------------------------------------------------------|
| 긴 함수                | `main()` 함수 내 모든 흐름이 포함                                        |
| 중복 코드              | `draw_chart()` 함수 여러 곳에 거의 동일한 시각화 코드                    |
| 부적절한 네이밍        | `fil`, `sent`, `kw` 등 문맥 없는 이름 사용                             |
| 전역 변수 사용         | 전역 상태로 감정 분석 결과 저장 (fil_data)                           |
| 매직 넘버/하드코딩     | 감정 키워드가 Constants에 리스트로 존재하되 중복되며 의미 불명확             |
| 테스트 미비            | 대부분의 함수에 대한 테스트 미구현                                      |

### 4.2. 안티패턴 목록 및 예시

| 안티패턴              | 반영 방식                                                                 |
|------------------------|--------------------------------------------------------------------------|
| God Function           | index.html UI, 데이터 로딩, 분석, 시각화 기능을 모두 수행               |
| Spaghetti Code         | 각 파일 간 의존성이 명확하지 않음                                       |
| Feature Envy           | Filter 함수가 외부 구조에 강하게 의존                              |
| Shotgun Surgery        | 키워드 카테고리를 변경하면 4~5개 파일에 수정 필요                        |
| Lava Flow              | 더 이상 쓰지 않는 File과 함수가 남아 있음 (FileHandler                   |

## 5. 예상 리팩토링 포인트 및 학습 가능한 리팩토링 기법

### 5.1. 주요 리팩토링 대상

| 리팩토링 대상        | 개선 필요 이유                                      |
|-----------------------|-----------------------------------------------------|
| index.html            | 많은 기능을 포함함 코드 분할 필요                     |
| TextAnalyzer          | 중복 제거 및 추상화 필요                           |
| FileHandler           | 파일 처리 기능의 분리 필요              |
| Session               | 전역 상태 → 상태 객체 또는 컨트롤러로 이동 필요   |
| 전역 상수             | 의미 있는 enum 또는 설정 클래스로 전환 필요        |

### 5.2. 적용 가능한 리팩토링 기법

| 리팩토링 기법                         | 적용 예시                                                           |
|---------------------------------------|----------------------------------------------------------------------|
| 함수 추출 (Extract Function)          | index.html 에서 SRP 적용하여 파일 분리                                 |
| 클래스 추출 (Extract Class)           | `FeedbackAnalyzer`, `FileManager`, `Visualizer` 등으로 분리         |
| 조건문 다형성 (Replace Conditional)   | 감정 분석 알고리즘 전략 패턴 적용                                  |
| 메서드 올리기 / 내리기                | 관련 메서드를 적절한 클래스로 이동                                 |
| 데이터 클래스 도입                    | 피드백 정보를 구조화된 클래스로 변경                               |
| 모듈화 및 디렉터리 리팩토링          | `components/`, `services/`, `models/` 디렉터리로 분리               |

## 6. 기대 학습 방향

### 6.1. 학습 가이드라인
1. 프로젝트 개요 소개 및 실습 준비 (1시간)
    - 전체 단계 미션 안내 
2. 테스트 구조 개선 및 case 추가 (2시간, 단계추가 필요) 
    - coverage 90% 이상
3. 프로젝트 코드의 오류 개선 (1.5시간)
    - log 표시가 page 에 level별로 출력 여부를 결정할 수 있도록 수정 (warning, error)
    - text 입력창이 multi line 입력을 받도록 수정
    - 동작오류 수정 - 필터 오류 수정("중립")
4. 네이밍 & 매직 넘버/하드코딩 & 전역 변수 사용 개선 (1시간)
5. 긴 함수 & 중복코드 개선 (1.5시간)
6. 리펙토링 관련 하나 추가 (1시간)
7. 추가 요구사항 구현 (3시간)
    - Trend 를 볼 수 있도록 시각화 (test_feedback_trend.csv 파일 사용)
    - 감정분석 필터의 관리를 위한 DB화(File DB 사용)
8. 팀 단위 리뷰 및 발표 (2시간)
    - 상대방 프로젝트 리뷰 및 장단점 정리 
    - 랜덤하게 발표
